**Document structure (exact sections, in this order):**

## 1. Encabezado y Metadatos
- Title: `# Auditoria Conceptual -- Sistema Inmobiliaria`
- Date, version, brief purpose statement

## 2. Proposito del Sistema (CONC-01)
- What the system is (ERP for real estate development companies)
- What business problem it solves
- Who uses it (internal staff, not end-clients)
- Core value proposition in 2-3 sentences

## 3. Dominio de Negocio (CONC-01)
- Explain the Argentine real estate development business model
- How loteos (subdivisions) work: a developer buys land, subdivides into lots, sells lots in installment plans
- The lifecycle: land acquisition -> development creation -> lot definition -> sales -> installment collection -> signing (escrituracion)
- How dual currency (USD/ARS) affects every transaction

## 4. Glosario de Terminos (CONC-01)
- Argentine real estate terms a non-Argentine developer would not know
- MUST include at minimum: Desarrollo, Lote, Manzana, Cuota, Refuerzo, Entrega, Cesion, Permuta, Escritura/Escrituracion, Boleto, Cotizacion, Caja, Cobranza, Proveedor, Vendedor (seller/comisionista)
- Format: term in bold, then definition, then how it maps to a code entity (e.g., "**Cuota** -- Pago periodico programado. En el sistema: modelo `Installment`")

## 5. Arquitectura de Alto Nivel (CONC-01)
- Stack: Next.js 15 App Router, TypeScript, PostgreSQL, Prisma ORM, Auth.js v5, shadcn/ui, Tailwind
- Layered architecture diagram in prose: Pages (src/app) -> Server Actions (src/server/actions) -> Services (src/server/services) -> Models (src/server/models) -> Prisma -> PostgreSQL
- Folder structure overview (from CLAUDE.md, but enriched with actual file counts discovered during reading)
- Key infrastructure: dual currency via dolarapi.com, email notifications, cron jobs, audit logging

## 6. Mapa de Modulos (CONC-02)
For EACH module below, write a subsection with:
- Module name and purpose (what business function it serves)
- Key pages (routes) in the dashboard
- Key server actions (list the exported functions from the actions file)
- Key models/services involved
- Business rules summary (what constraints/validations exist)

Modules to document (derive from navigation.ts groups + action files):
1. **Dashboard y Estadisticas** -- Landing page, KPIs, charts
2. **Desarrollos (Developments)** -- CRUD for real estate developments and their lots
3. **Personas (Persons)** -- Client/supplier management, client profile (ficha)
4. **Ventas (Sales)** -- Sale creation, installment plans, sale lifecycle (ACTIVA/CANCELADA/COMPLETADA/CONTADO/CESION)
5. **Cobranza (Collections)** -- Installment payment collection, pending payments view
6. **Caja (Cash/Treasury)** -- Cash movements, balance tracking, 14 movement types
7. **Firmas (Signings)** -- Signing slots, payment gate enforcement, commission auto-generation
8. **Mensajes y Notificaciones** -- Internal messaging, system alerts, notification bell
9. **Configuracion (Settings)** -- User management, role permissions, system config, business hours
10. **Auditoria (Audit Log)** -- Operation traceability, audit trail viewer
11. **Cotizacion (Exchange Rate)** -- Daily USD/ARS rate fetching and management
12. **Importacion (Import)** -- CSV/Excel/JSON data import
13. **Recibos (Payment Receipts)** -- Auto-generated payment receipts
14. **Tags** -- Tagging system for persons

## 7. Interconexion de Modulos (CONC-02)
- Prose description of how data flows through the system
- Data lifecycle: Development -> Lots -> Sale -> Installments -> Payments -> CashMovements -> Receipts
- Show dependencies: which modules depend on which (e.g., Sales depends on Persons, Developments, Lots, ExchangeRate)
- Summary table: Module | Depends On | Depended By | Key Data Flow

## 8. Modelo de Roles y Permisos (CONC-03)
- Explain the RBAC system: 4 roles, permission strings (resource:action format), DB-backed with hardcoded fallbacks
- Describe each role's intended business purpose:
  - SUPER_ADMIN: System owner, full access
  - ADMINISTRACION: Office staff managing sales and developments
  - FINANZAS: Finance team managing cash and payments
  - COBRANZA: Collections team, read-only view of sales and payments

## 9. Matriz de Acceso Completa (CONC-03)
- Full access matrix table: permissions as rows, roles as columns, checkmarks for access
- Extract EVERY permission from `src/lib/rbac.ts` DEFAULT_ROLE_PERMISSIONS
- For each permission, list what server actions/pages it gates
- Include both server-side guards (requirePermission calls in actions) AND UI-level gating (navigation.ts permission field, conditional rendering in pages)

## 10. Flujo de Autenticacion (CONC-03)
- Auth.js v5 setup: provider, session strategy, callbacks
- Middleware route protection (src/middleware.ts)
- Server-side auth guard pattern (requireAuth, requirePermission from auth-guard.ts)
- UI-level permission checking (sidebar filtering, conditional buttons)
- How permission DB override works (checkPermissionDb fallback to defaults)

**Writing guidelines:**
- Use layered depth: overview paragraph first, then details for each subsection

- Cross-reference between sections (e.g., "Ver seccion 6.4 Ventas para detalle del modulo")
- No diagrams -- use prose + tables as decided in CONTEXT.md
